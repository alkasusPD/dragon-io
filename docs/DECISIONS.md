# 의사결정 기록

중요한 결정과 그 이유를 남긴다. 방향이 바뀌면 이전 기록을 지우지 않고 새 결정을 추가한다.

## D-001 기술 기반

- 결정: HTML5, CSS, JavaScript, Canvas로 프로토타입을 만든다.
- 이유: 설치 없이 PC·모바일 브라우저에서 빠르게 검증할 수 있다.
- 상태: 유지

## D-002 화면 구성

- 결정: 9:16 세로형 탑뷰를 사용한다.
- 이유: 한 손 모바일 조작과 세로 진행형 슈팅에 적합하다.
- 상태: 유지

## D-003 아트 스타일

- 결정: 도트 스타일을 폐기하고 밝고 깔끔한 카툰 렌더링을 사용한다.
- 이유: 드래곤의 상품성과 모바일 화면 가독성을 높인다.
- 상태: 유지

## D-004 플레이어 캐릭터 규격

- 결정: 모든 구매형 드래곤은 프레임당 256×256px, 가로 12프레임 규격을 공유한다.
- 이유: 캐릭터 교체 비용을 낮추고 판정·애니메이션의 일관성을 유지한다.
- 상태: 애니메이션 품질 추가 개선 필요

## D-005 핵심 회피 시스템

- 결정: 하강은 3초 지속 후 자동 상승하며 2초 쿨타임을 갖는다.
- 이유: 상시 회피가 아닌 위험 패턴에 대응하는 제한형 스킬로 만든다.
- 상태: 플레이 테스트 필요

## D-006 적 디자인

- 결정: 적의 숫자보다 구분되는 공격 패턴과 예고를 우선한다.
- 이유: 화면 혼잡을 줄이고 하강 판단의 의미를 강화한다.
- 상태: 유지

## D-007 윙맨 성장

- 결정: LV.1 좌측, LV.2 우측 새끼 드래곤을 배치한다.
- 이유: 단계별 외형 변화가 명확하고 성장 체감이 즉각적이다.
- 상태: 유지

## D-008 BM 원칙 초안

- 결정: 플레이어 드래곤을 주요 수집·구매 대상으로 검토한다.
- 이유: 캐릭터의 시각적 매력과 표준 스프라이트 구조를 활용할 수 있다.
- 미결: 능력치 판매 범위, 가격, 획득 방식, 공정성 원칙

## D-009 메인 게임 플로우

- 결정: 실행 → 타이틀 → 로비(챕터 리스트) → 챕터 입장 → 웨이브 진행 → 결과 → 로비 구조를 사용한다.
- 이유: 플레이 콘텐츠 선택, 전투 준비, 결과 확인의 역할을 명확히 분리한다.
- 결과 분기: 클리어와 실패 모두 결과 화면을 거친다.
- 상태: 확정

## D-010 모바일 이동 조작

- 결정: 이동 입력은 화면 좌측 하단의 플로팅 조이스틱으로 제한하고, 플레이어는 화면 높이의 72% 아래로 내려가지 않게 한다.
- 이유: 손가락과 플레이어 드래곤을 분리해 캐릭터와 전방 탄막이 가려지는 문제를 줄인다.
- 변경: 조작 복잡도를 낮추기 위해 기존 하강 버튼과 하강 회피 시스템을 제거한다.
- 상태: 모바일 실기기 테스트 필요

## D-011 챕터 1 최종 보스

- 결정: 웨이브 10을 챕터 1의 마지막으로 하고 에메랄드 드레드윙 페이즈 2를 배치한다.
- 전투: 보라색 오라를 두른 상태에서 부채꼴 기본 구체와 2기의 느린 호밍 구체를 번갈아 발사한다.
- 완료 조건: 최종 보스 처치 즉시 챕터 클리어 결과 화면으로 전환한다.
- 상태: 첫 플레이 테스트 필요
